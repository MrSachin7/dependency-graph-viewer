| sm | 8px | Horizontal gap between toolbar buttons |

| lg | 24px | — (not used in toolbar) |
| xl | 32px | — (not used in toolbar) |
| 2xl | 48px | — (not used in toolbar) |

Exceptions:

| Token | Value | Justification |

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label | 12px | 500 (medium) | 1.2 |
| Button text | 13px | 500 (medium) | 1 |

Notes:
- Body (14px / 400) matches `text-sm` used throughout InfoPanel.
- Label (12px / 500) matches `text-xs font-medium` pattern used for field labels and type badges in existing nodes.
- Button text (13px / 500) matches `text-[13px] font-medium` used on node labels — keeps toolbar visually consistent with graph elements.
- Geist Sans used for all text; no mono in toolbar.
- Two weights only: 400 (regular) for body copy, 500 (medium) for all interactive and labeling elements within the toolbar.

Source: ActivityNode.tsx (`text-[13px] font-medium`), ResourceNode.tsx (`text-[10px] font-medium` for badges), InfoPanel.tsx (`text-sm`, `text-xs font-medium`, `text-lg font-semibold`).

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#f6f4ee` | Page background, canvas area background |
| Secondary (30%) | `#ffffff` | Toolbar background, InfoPanel sidebar, node backgrounds |
| Accent (10%) | `#2563eb` (blue-600) | Active sort toggle background + label, reload button hover state |
| Destructive | — | Not applicable — no destructive actions in this phase |

Accent reserved for: active sort toggle fill (background + text) only. The reload button uses a neutral/outlined style — it is not a destructive action and does not warrant accent color.

Additional semantic colors (pre-existing, not assigned new usage in this phase):
- `#16a34a` (green-600): resource node borders (do not reuse on toolbar)
- `#f59e0b` (amber-400): SPOF ring decoration (do not reuse on toolbar)

**Toolbar surface:** `#ffffff` (white) — matches the InfoPanel sidebar. The toolbar sits above the warm `#f6f4ee` canvas; white creates a clean horizontal separation without adding a new color.

**Visual hierarchy:** The graph canvas is the primary visual anchor. The toolbar is secondary chrome — no element within the toolbar should compete for attention with the graph.

Source: globals.css (`--background: #f6f4ee`), InfoPanel.tsx (`bg-white`, `border-l border-gray-200`), ActivityNode.tsx (`border-blue-600`). CONTEXT.md D-09 (sort toggles show active/inactive visual state). CONTEXT.md discretion area (toolbar background: neutral consistent with existing graph background).

## Component Inventory

### Toolbar (new: `src/components/Toolbar.tsx`)

**Layout:** Full-width horizontal bar, 44px height, white background, `border-b border-gray-200` bottom border to separate from canvas. Three controls left-aligned with 8px gap between them, 16px left padding.

**Controls left-to-right:**
1. **Reload button** — outlined style, neutral. Label: "Reload". 13px / 500 weight. Border: `border border-gray-300`, background: transparent. Hover: `bg-gray-50`. No active/inactive concept — it is a momentary action. Height: 32px. Horizontal padding: 12px.
2. **Sort Activities toggle** — two-state. Inactive: outlined (`border border-gray-300`, transparent bg, `text-gray-600`). Active: filled (`bg-blue-600`, `text-white`, no border). Label: "Sort Activities". Same sizing as Reload button. State indicator: filled blue background (no separate icon needed).
3. **Sort Resources toggle** — identical behavior and sizing to Sort Activities toggle. Label: "Sort Resources".

**Props interface:**
```
{
  onReload: () => void;
  sortActivities: boolean;
  onToggleSortActivities: () => void;
  sortResources: boolean;
  onToggleSortResources: () => void;
}
```

Source: CONTEXT.md D-09 (three controls in a row), D-10 (Toolbar.tsx new component, props from GraphCanvas), CONTEXT.md code_context integration points. Discretion applied: toolbar bg white, active = filled blue-600, button height 32px within 44px bar.

## Interaction Contract

### Reload Button
- Click → calls `router.refresh()` via `useRouter` from `next/navigation`
- Effect: server component re-runs, fresh `initialNodes`/`initialEdges` props flow down, all state resets (selection, sort toggles, layout)
- No loading spinner required (Next.js router refresh is fast for static JSON)
- Button does not change appearance during reload

Source: CONTEXT.md D-01, D-02.

### Sort Activities Toggle
- Click when OFF → sets `sortActivities = true` → node list re-sorted: `critical (0) → high (1) → medium (2) → low (3)` → dagre re-runs → nodes reposition
- Click when ON → sets `sortActivities = false` → original `initialNodes` order restored → dagre re-runs
- Visual: inactive = outlined gray button; active = filled blue-600 button
- Both sort toggles are independent; both can be ON simultaneously

Source: CONTEXT.md D-03, D-04, D-06, D-07.

### Sort Resources Toggle
- Click when OFF → sets `sortResources = true` → node list re-sorted: SPOF nodes first (isSPOF === true), then by dependencyCount descending within each group → dagre re-runs
- Click when ON → sets `sortResources = false` → original order restored → dagre re-runs
- Visual: same active/inactive pattern as Sort Activities

Source: CONTEXT.md D-03, D-05, D-06, D-07.

### Canvas Area Adjustment
- Without toolbar: `height: 100vh`
- With toolbar (Phase 4): canvas flex column; toolbar takes 44px; canvas area takes remaining height via flex-grow or `calc(100vh - 44px)`

Source: CONTEXT.md D-08.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | "Reload" (verb only — single clear action, no noun needed) |
| Sort Activities toggle (inactive) | "Sort Activities" |
| Sort Activities toggle (active) | "Sort Activities" (same label — visual fill communicates state, label stays constant) |
| Sort Resources toggle (inactive) | "Sort Resources" |
| Sort Resources toggle (active) | "Sort Resources" (same constant label) |
| Empty state heading | Not applicable — toolbar is always visible regardless of graph state |
| Empty state body | Not applicable |
| Error state | Not applicable — reload failure is handled by Next.js router, no custom error UI in scope |
| Destructive confirmation | None — no destructive actions in this phase |

Rationale for constant toggle labels: changing the label on toggle (e.g., "Unsort Activities") creates cognitive overhead. Consistent labels + visual fill state is the established pattern (e.g., GitHub filter toggles, Linear priority filters).

Source: CONTEXT.md D-09 (sort toggles show active/inactive visual state). REQUIREMENTS.md CTRL-01, CTRL-02, CTRL-03.
