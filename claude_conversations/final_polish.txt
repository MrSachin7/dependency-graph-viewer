⏺ Honestly, for this specific app, I'd push back a little — and here's why:

  The abstraction already exists — it's page.tsx.

  page.tsx is the only place that knows where data comes from. GraphCanvas
  already receives initialNodes and initialEdges as props — it has no idea
  whether those came from a file, an API, or a hardcoded fixture.
  transformGraphData in graph.ts only knows about GraphJson shape, not the
  source. So the data/transport concern is already isolated to one 4-line file.

  The pattern you're describing (a getGraphData() fetcher) would help if:
  - Multiple pages/routes need the same data
  - You're switching between implementations during development (mock vs. real
  API)
  - You have server-side vs. client-side fetching that needs to stay
  interchangeable

  None of those apply here:
  - One page, one data source, one call site
  - The "switch to backend" scenario would be: delete readFileSync, add
  fetch('/api/graph'), done — the change is tiny and local to page.tsx
  - Adding a src/data/graphFetcher.ts with a getGraphData() that wraps the same
  readFileSync call just moves 2 lines into a new file without adding any real
  indirection

  The one case where I'd add it: if you're planning to add a Route Handler
  (/api/graph) and GraphCanvas's reload button needs to re-fetch client-side.
  Right now handleReload uses router.refresh() which re-runs the server
  component — that works, but a proper client-side fetch('/api/graph') would be
  faster and more explicit. In that case, a src/lib/fetchGraph.ts (not a whole
  directory) that exports one async function used by both the API route and the
  client hook would be worth having.

  Short answer: the current structure is appropriate for the current scope. The
  abstraction you're describing would be premature here — not because the idea
  is wrong, but because page.tsx is that boundary already.
